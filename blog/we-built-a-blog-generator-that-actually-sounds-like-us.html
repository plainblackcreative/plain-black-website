We built a blog generator that doesn't sound like ChatGPT

6 min read By PlainBlack 10 May 2026
← Back to Blog

I had a confession in the week-1 report on Sunday: I was the bottleneck. Not the build. The blog. Writing a thousand-word post in our voice every morning, with no help from a generic AI, while shipping a tool by lunchtime, turned out to be the part of the math I had not quite worked out before signing up to a daily-cadence challenge. By Sunday night it was clear that if we did not fix the input side, we would run out of fuel before week three.

Today I built the fix.

The short version

One topic in. Four outputs out. A blog post in our voice (not generic AI), a Facebook post, a LinkedIn post, image prompts for whatever generates your hero, alt text, and tags. All tuned to the PlainBlack brand. The blog body uses our actual component library, callouts and post-images and dividers, not just plain HTML. Drafts get a shareable URL but stay off Google. The whole content lifecycle (Draft, Published, Archived, Deleted, with one-click recovery from git history) is in one tool because most blog tools stop at "draft to publish" and call it done.

It is an internal tool. We use it. Day 10 of 30 ships it. From day 11 onwards it carries half the load of writing this challenge.

Try it

This very post was generated through the tool. So was our response to Namecheap's piece on whether blogging is still relevant. Read both. Decide if either reads like AI slop.

What it does, three things that matter

One. Voice. Not generic AI.

Most "AI content tools" wire up GPT or Claude with a vague "write a blog about X" prompt and call it shipping. The output reads like every other AI-written blog you have scrolled past this year. We built the prompt from our actual brand kit: voice rules, tone choices, the specific banned phrases ("In a world where...", "What do you think?", "Hot take:", and the rest of the LinkedInfluencer cliché set), and the platforms' real differences. Facebook gets the casual, observational, mate-share-what-worked tone. LinkedIn gets the sober, insight-led, prose-led version. They are explicitly told to differ from each other; if the same draft would work for both, the tool is told to rewrite. The blog body uses our component library, so callouts and post-images come out tagged correctly without hand-editing.

The point is not speed. ChatGPT is fast. The point is the output sounds like us instead of like a content marketer doing a "value-add post." That is the whole reason this thing exists.

Two. The full lifecycle, not just draft and publish.

Most blog tools stop at "draft to publish." We added the rest:

  • Draft. File pushed to /blog/{slug}.html with a noindex tag, off the grid, off the sitemap. The URL works (you can send a preview link to someone), but Google does not index it. This is what the two posts you are reading right now are: shareable, but unlisted.
  • Published. The obvious one. On /blog, in sitemap, indexed.
  • Archived. The file stays at /blog/{slug}.html. Old archive flows moved files, breaking inbound links from anyone who shared the post. Ours does not. Just adds noindex, off the grid, off the sitemap. Restore is one click.
  • Deleted. File gone, image siblings gone, library entry kept as a tombstone (slug + title + date + category + the social copy you wrote). The slug stays reserved; if you try to write a new post that derives the same slug, the tool warns explicitly because the URL had a meaning before.
  • Recover. Tombstones get a Recover button. It walks the GitHub commits API to find the parent of the delete commit, pulls the post HTML and every asset sibling that existed at that ref, and restores the lot. One click. Closes the "wait, no I changed my mind" door from being a one-way door.

All of that is tag-driven on the public side. The 30-day challenge filter pill at the top of /blog reads tags, so a post can have any sensible category alongside the challenge tag. Decoupled, on purpose, so once the challenge ends the posts wear a permanent label that still makes sense.

Three. Tag dictionary so the same idea is not three different tags.

Let any AI tool generate tags from scratch every time and you end up with 30-day-challenge AND 30 day challenge AND challenge AND the-30-day-challenge as four separate things. We seed the prompt with every tag already in use across the blog library, tell the model to reuse them where they fit, and only invent new ones when nothing fits. Result: when someone clicks the marketing tag on /blog, they get every post tagged marketing, not just the 30% the AI happened to spell that way today.

Smaller things that earn their keep

  • Quick HTML edit for surgical fixes. Typo, broken link, rate change in a quote tool post. Edit the rendered HTML in a collapsible textarea inside the Publish tab, click Update Post. No LLM call, no metadata regeneration. The Facebook copy you spent ten minutes polishing yesterday stays exactly as it was.
  • Save as Draft with a shareable URL. Useful when you want a colleague to review a post before it goes live without it appearing on /blog or in Google.
  • Image specs as a modal. Hero needs to be 1200 by 630 for the social card to crop right. Click the help dot, get the spec. Don't hunt through documentation.
  • Auto-context-aware alt text. The model reads the body and proposes alt text per image based on what the post is actually about. Pre-populates the alt field on each image preview card. Override when you want; the default is something useful instead of "image".
  • Quality check. Before publish, an inline panel scores the post: title length, meta description, word count, heading structure, image alt coverage, internal link presence, tag count, link hygiene (catches http://, internal .html suffixes that should be clean URLs, broken #anchor refs).

What is missing (and shipping next)

This is day 10. The version that is live today is good enough to carry the daily cadence. The version coming in the next pass adds:

  • Per-field regenerate. Tick the checkbox next to Title, Body, Excerpt, Facebook, LinkedIn, Tags, Alt, or Image Prompts. Click Regenerate. Only the ticked fields rerun through the model. The rest stay verbatim. Today, hitting Update Preview re-runs everything in preserve mode, which is fine but heavier than it needs to be, and a real risk if you have polished the LinkedIn copy.
  • Editable fields on the left. Today, fixing the Facebook copy means going through Library, then Edit, then Quick HTML edit, or regenerating the whole post. The next pass moves Facebook, LinkedIn, Excerpt, and Alt text to first-class fields on the left column with their own checkboxes.

That is the difference between "shipped" and "polished." The shipped version writes posts. The polished version is precise.

For The Nerds details

A little more on what is actually under the bonnet.

  • Single static page. The whole admin tool is one HTML file at admin.plainblackcreative.com, around five thousand lines. No build step, no framework. Cloudflare Pages deploys it in roughly thirty seconds after a commit. The admin's JavaScript runs in the browser and uses the user's GitHub PAT (from localStorage) to make Contents API calls directly.
  • Library is a JSON file. docs/blog-library.json. Holds every post's metadata. Read on admin load, written via Contents API on every state change. Old entries that pre-date the v2 schema fall through gracefully.
  • The tag dictionary is reduced from the library on each generation. It is not a separate file. The prompt sees TAG DICTIONARY: 30-day-challenge, ai-tools, branding, blogging, ... as part of the instruction set, so the model knows what canonical strings exist before it picks tags for the new post.
  • Save as Draft does an in-place HTML edit. Inject a <meta name="robots" content="noindex"> right after the <meta charset>. On Restore (archived to published) or Go Live (draft to published), the tag is stripped back out cleanly. Round-trip tested.
  • Recovery from git history uses two GitHub APIs. The Commits API finds the parent of the delete commit. The Contents API at ?ref={parentSha} pulls the file content. For files over 1MB the Contents API silently returns empty content, so we fetch via raw.githubusercontent.com/{repo}/{ref}/{path} and base64-encode locally for the PUT. Without that fallback, recovery silently commits 0-byte files; exactly the bug we hit and fixed within an hour today.
  • Asset cleanup on delete probes 40 candidate paths (slot 1 through 10, times webp, png, jpg, jpeg) in parallel via Promise.all. Sequential probing left the confirmation dialog frozen for 30 seconds while the user wondered if the button worked.
  • The voice prompts include explicit banned phrases. "Hot take:", "What do you think?", "In a world where...", "I learned something today", "Drop a comment below"; all named, all blocked. Plus a clause: "if your draft for one would also work for the other, rewrite." Forces Facebook and LinkedIn to actually diverge, not be the same paragraph with different hashtags.
  • Voice-first, model-second. Switch the underlying LLM and the prompt structure means the brand voice survives the swap. Today it runs through Claude. Tomorrow it could run through anything that takes structured prompts. The voice rules live in our prompt, not in the model.
